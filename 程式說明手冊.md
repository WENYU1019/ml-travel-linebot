## 核心執行流程

```text
app.py
  |
  v
analysis.engine.analyze_dialogue(text)
  |
  +--> extraction.extractor.extract_info(text)
  |
  +--> analysis.llm_judge.judge_with_llm(text, info)
  |
  +--> analysis.classifier.classify(text, info)
  |
  v
data.models.AnalysisResult
```

`analysis.engine` 是核心協調者。它會先呼叫 `extract_info()` 擷取對話資訊，再優先呼叫 `judge_with_llm()`。如果 Groq API 不可用、金鑰缺失或 LLM 回傳格式錯誤，就改用 `classify()` 做規則式 fallback。

## app.py

用途：

- 建立 Flask app。
- 載入 `.env`。
- 建立 LINE Bot client。
- 提供 `GET /` 健康檢查。
- 提供 `POST /callback` 給 LINE webhook。
- 收到文字訊息後呼叫 `analyze_dialogue()`。

主要調用：

```python
from analysis.engine import analyze_dialogue

result = analyze_dialogue(event.message.text)
```

回覆邏輯：

- `should_intervene = false`：不回覆。
- `should_intervene = true` 且 `requires_external_search = false`：回覆 `suggested_reply`。
- `should_intervene = true` 且 `requires_external_search = true`：先回覆 `intermediate_reply`。

## analysis/

### analysis/engine.py

核心入口：

```python
from analysis.engine import analyze_dialogue

result = analyze_dialogue(text)
payload = result.to_dict()
```

責任：

- 呼叫資訊擷取。
- 呼叫 LLM 判斷。
- LLM 失敗時切換 fallback。
- 統一回傳 `AnalysisResult`。
- 判斷是否需要外部搜尋。

### analysis/llm_judge.py

責任：

- 讀取 `GROQ_API_KEY` 和 `GROQ_MODEL`。
- 呼叫 Groq Chat Completions API。
- 要求 LLM 依專案格式回傳 JSON。
- 將 LLM 回傳轉成 `AnalysisResult`。
- 失敗時丟出 `LLMJudgeError`，讓 `engine.py` 接手 fallback。

### analysis/classifier.py

責任：

- 規則式情境分類。
- 當 LLM 不可用時仍能輸出基本分析。
- 依 `data.knowledge_base.SCENARIOS` 的關鍵字與 feature hints 計算分數。

## extraction/

### extraction/extractor.py

核心入口：

```python
from extraction.extractor import extract_info

info = extract_info(text)
```

責任：

- 從文字中擷取：
  - `time`
  - `location`
  - `people_count`
  - `budget`
  - `constraints`
  - `activity_types`
  - `options`
  - `decision_state`
  - `risk_info`
  - `need_type`

輸出型別是 `data.models.ExtractedInfo`。

## data/

### data/models.py

主要資料結構：

- `ExtractedInfo`：對話中擷取出的資訊。
- `ScenarioDefinition`：情境規則與建議行為。
- `AnalysisResult`：完整分析結果。

常用方法：

```python
payload = result.to_dict()
```

### data/knowledge_base.py

責任：

- 存放可判斷的情境資料。
- 提供情境代碼、名稱、關鍵字、feature hints、系統行為與建議回覆。
- 給 `classifier.py` 與 `llm_judge.py` 使用。

## AnalysisResult 輸出欄位

`analyze_dialogue(text).to_dict()` 會產生：

- `scenario_code`：情境代碼。
- `scenario_name`：情境名稱。
- `stage`：目前討論階段。
- `should_intervene`：是否應該介入。
- `intervention_type`：介入類型。
- `confidence_score`：信心分數。
- `evidence`：判斷依據。
- `system_behavior`：系統行為說明。
- `requires_external_search`：是否需要外部搜尋。
- `intermediate_reply`：需要搜尋時先回覆的文字。
- `suggested_reply`：建議回覆內容。
- `extracted_info`：擷取出的對話資訊。

## 新增情境規則

要新增一種判斷情境，主要修改：

1. `data/knowledge_base.py`
2. `analysis/classifier.py`，如果新情境需要新的規則計分。
3. `analysis/llm_judge.py`，如果 LLM prompt 或輸出格式需要同步調整。

新增後請用 CLI 測試：

```powershell
python -m interfaces.cli --input sample_dialogue.txt
```

## 新增擷取欄位

如果要新增新的對話資訊欄位：

1. 在 `data/models.py` 的 `ExtractedInfo` 加欄位。
2. 在 `extraction/extractor.py` 加擷取邏輯。
3. 視需求更新 `analysis/llm_judge.py` 的 prompt。
4. 更新本文件的 `AnalysisResult` 或 `ExtractedInfo` 說明。

## 維護注意事項

- `app.py` 只做 LINE Bot 入口與回覆流程，不放大量判斷規則。
- 對話判斷放 `analysis/`。
- 資訊擷取放 `extraction/`。
- 資料模型與情境知識庫放 `data/`。
- CLI 和整合範例放 `interfaces/`。
- 不要恢復 `ai_linebot_core/` 子專案資料夾。
- 不要在子資料夾建立新的 venv。
- 真實 API key 只放 `.env`。
